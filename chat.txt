● Here are the key performance differences between robosync on Linux and Windows:

  File Copy Implementation

  Linux:

  - These are zero-copy operations - data transfers directly between file descriptors without going
  through userspace
  - Very efficient for local operations

  Windows:
  - fs::copy() uses CopyFileW() API
  - Goes through Windows file system layers with more overhead
  - Not optimized for network transfers like RoboCopy's internal APIs

  Network Performance

  Linux:
  - Better performance over network mounts (NFS, CIFS)
  - Kernel-level optimizations for network file systems
  - sendfile() works efficiently with network sockets

  Windows:
  - SMB/CIFS has more overhead through standard APIs
  - RoboCopy uses undocumented SMB optimizations that fs::copy() doesn't access
  - Network drives (like H:) suffer from synchronous I/O bottlenecks

  Threading Efficiency

  Linux:
  - Lower thread creation overhead
  - Better scaling with high thread counts
  - File operations don't block as heavily

  Windows:
  - Higher per-thread overhead
  - File handles and synchronization more expensive
  - Each fs::copy() blocks the thread completely

  I/O Patterns

  Linux:
  // Linux fs::copy pseudocode
  copy_file_range(src_fd, dest_fd, len) // Zero-copy
  // or
  sendfile(dest_fd, src_fd, offset, count) // Zero-copy

  Windows:
  // Windows fs::copy pseudocode
  CopyFileW(src_path, dest_path, fail_if_exists) // Synchronous

  Linux (local disk):
  - Can achieve 1-2 GB/s easily
  - Scales well with thread count
  - Network: 100-500 MB/s typical

  Windows (local disk):
  - Usually 200-500 MB/s
  - Doesn't scale as well with threads

  RoboCopy (Windows only):
  - 8-9 GB/s (using special APIs)
  - Asynchronous I/O
  - SMB multichannel
  - Kernel-level optimizations

  2. Windows philosophy: Hide complexity behind high-level APIs
  3. RoboCopy secret sauce: Uses undocumented NT kernel APIs, not available to normal programs
